# XD 下一轮改进建议清单 Stage35

## A. 立即可做，不需要新 API

1. 报告措辞优化  
   - 把“样本口径”提示放得更靠前、更醒目
2. 业务摘要格式优化  
   - 把“今天最值得看 3 个点”固定成顶部模块
3. 老板摘要继续压缩  
   - 控制在更容易转发的一页以内
4. 异常项高亮  
   - 把 `unpay`、`undeliver`、`FAIL_*`、`PASS_NO_DATA` 做成更显眼的摘要格式
5. 每日 / 每周固定输出  
   - 明确每天和每周的固定生成节奏
6. 试用包与汇报包模板化  
   - 让业务侧和管理层看到的格式保持一致

## B. 需要业务反馈后再做

1. 指标命名调整  
   - 看业务是否更容易理解“订单样本数”“接口信号”“窗口命中数”等命名
2. 看板字段排序  
   - 根据业务使用习惯调整“订单优先”还是“商品优先”
3. 输出格式调整  
   - 看业务更偏好 Markdown、飞书、Excel 还是邮件摘要
4. 高频问题字段补解释  
   - 哪些字段最容易误读，就优先做解释强化
5. 每日 / 每周摘要长度调整  
   - 看业务更偏好极简版还是详细版
6. 运营摘要聚焦点优化  
   - 看业务更在意异常提醒、商品抽查，还是订单状态堆积

## C. 需要外部新证据或新数据口径后才能做

1. 全量 GMV
2. 转化率
3. 国家结构
4. 产品贡献
5. 完整经营诊断
6. 严格窗口全量订单数
7. 严格窗口全量商品变化数
8. restriction 对象重开
9. 任何需要新 live 证据才能证明的对象级能力

## 当前建议的优先级

当前下一轮最合理的顺序是：

1. 先收业务反馈
2. 再做表达和格式优化
3. 最后才考虑是否需要新的外部证据来支持更深层的数据能力

## 当前明确不建议做的事

- 回头继续 API 同构重试
- 把当前页样本写成全量经营结果
- 在没有新证据的情况下重开 restriction 对象
- 为了追求“更完整”去扩写侧或高风险区
