# WIKA 运营报告写作规范

## 目标

本规范只解决一件事：让 WIKA 产出的运营报告真正能被业务方拿去开会、排优先级、派动作、追结果。

报告不是系统能力说明，不是 JSON dump，不是字段清单。
报告必须把真实数据转成经营判断，再转成优先行动。

## 目标读者

### 老板 / 管理层
- 只关心整体状况、主要风险、最值得先做的动作。
- 不需要看所有字段，但需要知道判断依据和盲区。

### 运营负责人
- 需要知道当前最严重的问题在哪一层。
- 需要知道本周优先级怎么排、先抓哪几个点。

### 销售 / 跟单 / 执行同事
- 需要知道哪些动作要自己接手。
- 需要知道 WIKA 已经帮他整理到哪一步，还缺什么输入。

## 一份有用报告必须解决的问题

正式运营报告必须明确回答：
- 现在整体状况如何。
- 哪些指标最值得关注。
- 本周 / 本周期是变好还是变坏。
- 为什么会这样。
- 先做什么最值。
- 哪些事情 WIKA 可以直接支撑。
- 哪些事情必须人工接手。

## 固定四层结构

每一份正式报告都必须严格区分以下四层，不能混写。

### 第一层：事实层
- 只写真实数据、真实变化、真实已确认字段。
- 只能使用当前 official mainline confirmed 字段，或明确标注为 derived 的内容。
- 不能把 route 原始 JSON 直接塞进正文。

### 第二层：判断层
- 所有判断都必须由事实支撑。
- 不能只写“可能”“也许”“大概”，必须明确写出判断依据。
- 如果证据不足，只能写“当前数据不足以证明”，不能硬下结论。

### 第三层：建议层
- 每条建议必须可执行、可排序、可交接。
- 必须写清：做什么、为什么、预期收益、谁执行、WIKA 当前支持到什么程度。
- 所有建议都必须按优先级排序，至少分为 `P1`、`P2`、`P3`。

### 第四层：盲区层
- 必须明确哪些结论受到数据盲区限制。
- unavailable 维度不能沉默省略。
- 盲区只写真正影响判断的关键项，不要把所有 unavailable 字段机械堆满正文。

## 报告必须回答的固定问题

每份正式运营报告必须显式回答以下五个问题：
1. 当前最重要的 3 个发现是什么。
2. 当前最危险的 3 个问题是什么。
3. 当前最值得先做的 3 个动作是什么。
4. 哪些建议依赖人工执行。
5. 哪些判断受数据盲区限制。

## 执行摘要规则

执行摘要必须满足：
- 只允许 5–8 行。
- 先写结论，不先铺背景。
- 第一行必须能回答“现在整体状况如何”。
- 至少包含一个机会点、一个风险点、一个最优先动作。
- 不允许出现长字段串、路由列表或 JSON 片段。

## 核心发现规则

核心发现必须满足：
- 最多 3–5 条。
- 每条都要写清：
  - 发现
  - 证据
  - 影响
- 发现是“经营现象”，不是“字段存在”。

错误写法示例：
- “当前有 `visitor / imps / clk / reply`。”

正确写法示例：
- “当前店铺层流量没有塌陷，但来源结构不可见，因此只能确认整体活跃度，不能确认是哪类渠道在驱动增长。”

## 关键问题规则

关键问题必须满足：
- 最多 3–5 条。
- 每条都要写清：
  - 问题
  - 为什么重要
  - 如果不处理会怎样
- 问题必须能和行动项对上，不允许只报问题不落动作。

## 优先行动规则

优先行动必须满足：
- 按 `P1`、`P2`、`P3` 输出。
- 每条都要写清：
  - 做什么
  - 为什么
  - 预期收益
  - 需要谁执行
  - WIKA 当前能支持到什么程度

优先级含义固定为：
- `P1`：立即做
- `P2`：本周做
- `P3`：后续跟进

## 盲区表达规则

盲区部分必须满足：
- 只列关键盲区。
- 每个盲区要说明它影响什么判断。
- 必须清楚区分：
  - 当前没有
  - 当前有但只是 derived
  - 当前有 historical / legacy evidence，但不能当 current official mainline confirmed

## 边界声明规则

每份正式报告都必须单独有“边界声明”章节，至少说明：
- 哪些结论是 official confirmed。
- 哪些结论是 derived。
- 哪些维度仍 unavailable。
- 当前不是完整经营驾驶舱。
- task3 / task4 / task5 当前不是平台内闭环。
- task6 不在当前范围内。

## 绝对禁止的写法

正式报告禁止出现以下写法：
- 罗列所有字段。
- 把 route / JSON 当成正文。
- 大段堆 unavailable 字段而不解释影响。
- 没有优先级排序。
- 没有 next action。
- 没有人工接手说明。
- 把系统能力说明混进经营结论。
- 把 derived 当 official。
- 把 unavailable 沉默省略。
- 为了显得全面，把正文写成流水账。
- 把技术细节写得比经营判断还多。

## 排版要求

报告必须满足以下排版要求：
- 一屏内先看到结论，再看到证据，再看到动作。
- 标题层级固定，避免层级过深。
- 一条发现尽量控制在 3 行内讲清。
- 一条行动尽量控制在 5 行内讲清。
- 不使用大段原始表格作为正文主体。
- 不把 evidence 路径、脚本路径、字段全名堆到正文开头。

## 生成顺序要求

报告生成器必须按以下顺序工作：
1. 汇总真实数据。
2. 提炼事实层。
3. 形成判断层。
4. 生成行动候选。
5. 排序优先级。
6. 补盲区与边界声明。
7. 最后填入正式模板。

禁止反过来先拿 route 输出拼正文。

## 交付门槛

一份正式可交付的 WIKA 运营报告，最低要满足：
- 有执行摘要。
- 有核心发现。
- 有关键问题。
- 有优先行动。
- 有盲区。
- 有边界声明。
- 有人工接手说明。
- 不把 derived 写成 official。
- 不把报告正文写成系统输出堆砌。

达不到上述门槛的报告，只能算调试产物，不能交付业务方。
