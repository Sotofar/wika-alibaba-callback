# WIKA 运营报告写作规范

## 目标

本规范只解决一件事：让 WIKA 产出的报告服务经营决策，而不是服务系统炫技。

一份可交付的 WIKA 运营报告，必须让读者在几分钟内看清：
- 现在整体状况如何
- 最严重的问题是什么
- 先做哪三件事
- 哪些动作 WIKA 能支撑
- 哪些动作必须人工接手
- 哪些判断受盲区限制

## 当前报告失败的根因

当前不合格报告通常有以下共性：
1. 报告是系统视角，不是运营视角。
2. 报告是数据堆砌，不是经营判断。
3. 报告是字段罗列，不是优先级排序。
4. 报告是技术摘要，不是执行清单。
5. 报告没有严格区分事实、判断、建议、盲区。
6. 报告没有告诉读者先做哪三件事、由谁执行、哪里必须人工接手。
7. 报告排版不适合开会，不适合老板、运营、销售直接使用。

本规范的所有要求，都是为了解决这七个根因。

## 目标读者

### 老板 / 管理层
- 只关心整体判断、主要风险、最优先动作和需要拍板的事项。
- 不需要看全量字段，但必须知道证据来源和盲区边界。

### 运营负责人
- 需要知道问题位于店铺、产品、订单、广告还是执行链路。
- 需要知道本周资源怎么排、先抓哪几个动作最值。

### 店铺运营 / 产品运营
- 需要知道哪些商品、哪些内容、哪些结构需要先改。
- 需要看到可执行动作，而不是字段清单。

### 销售 / 跟单 / 人工接手人员
- 需要知道哪些上下文缺失、哪些信息必须补齐、交接到哪一步。
- 需要明确系统已经准备好的内容和自己必须补的内容。

## 一份有用的运营报告必须回答的问题

每份正式报告必须明确回答：
1. 现在整体表现如何。
2. 本周期最重要的 3 个发现是什么。
3. 最严重的 3 个问题是什么。
4. 最值得先做的 3 个动作是什么。
5. 哪些动作 WIKA 可以直接支撑。
6. 哪些动作必须人工完成。
7. 哪些判断受数据盲区限制。

## 固定四层结构

每份正式报告都必须严格区分四层，不能混写。

### 第一层：事实层
- 只写真实 route 结果、真实字段、真实变化、真实已确认输入。
- official 和 derived 必须分开标注。
- degraded route 只能写成“降级参与”，不能冒充 full success。
- 禁止把 route 原始 JSON 直接塞进正文。

### 第二层：判断层
- 每个判断都必须能追溯到事实层。
- 判断必须回答“为什么重要”，不能只写现象。
- 如果证据不足，只能写“当前数据不足以证明”。

### 第三层：建议层
- 每条建议必须写清：
  - 做什么
  - 为什么
  - 预期收益
  - 执行人
  - WIKA 当前能支撑到什么程度
  - 是否需要人工确认
- 建议必须按 `P1 / P2 / P3` 排序。

### 第四层：盲区层
- 只写真正影响判断的关键盲区。
- 每个盲区必须说明它影响了什么判断。
- unavailable 维度不能沉默省略。

## 报告的固定结构顺序

正式报告固定按以下顺序组织：
1. 执行摘要
2. 核心发现
3. 关键问题
4. 优先行动
5. 分层诊断
6. 工作量替代评估
7. 数据盲区
8. 路由健康与证据约束
9. 边界声明

禁止先铺背景、先讲系统能力、先列 route。

## 执行摘要规则

执行摘要必须满足：
- 只允许 5–8 行。
- 第一行必须先下总判断。
- 必须至少覆盖：
  - 一个最重要发现
  - 一个最严重问题
  - 一个最优先动作
  - 一个最关键盲区
- 禁止出现大段字段名、路由列表、JSON 片段、脚本路径。

## 核心发现规则

核心发现必须满足：
- 最多 3–5 条。
- 每条必须包含：
  - 发现
  - 证据
  - 影响
- 发现必须是经营现象，不是字段存在说明。

错误写法：
- “当前有 `visitor / imps / clk / reply`。”

正确写法：
- “店铺层活跃度仍可读，但来源维度缺失，因此当前只能确认整体没有塌陷，不能确认是哪类渠道在驱动变化。”

## 关键问题规则

关键问题必须满足：
- 最多 3–5 条。
- 每条必须包含：
  - 问题
  - 为什么重要
  - 如果不处理会怎样
- 每个问题都必须能映射到至少一个动作。

## 优先行动规则

优先行动必须满足：
- 按 `P1 / P2 / P3` 输出。
- 每条必须包含：
  - 做什么
  - 为什么
  - 预期收益
  - 执行人
  - WIKA 当前能支持到什么程度
  - 是否需要人工确认
- 不允许出现“继续观察”“持续关注”这类空动作，除非同时给出观察对象、责任人和下一步触发条件。

优先级含义固定为：
- `P1`：立即做
- `P2`：本周做
- `P3`：后续跟进

## 路由健康与证据使用规则

报告生成时必须先判断 route 健康度：
- `full_success`
- `degraded`
- `failed`

如果高层 route 为 `degraded`：
- 允许保留其排序或摘要能力
- 但核心事实必须回退到更底层稳定 route
- 正文必须明确写出哪些高层 route 只是降级参与

如果 route `failed`：
- 不得把旧 evidence 写成当前 live 成功
- 可以在证据层记录失败
- 只有在底层稳定数据仍足够时，才允许继续生成报告

## 绝对禁止的写法

正式报告禁止出现以下写法：
- 把 JSON / route 输出原样拼进正文。
- 堆字段、堆 unavailable、堆脚本路径。
- 用技术日志口吻替代经营判断。
- 没有优先级。
- 没有执行人。
- 没有人工接手说明。
- 没有盲区说明。
- 把 derived 当 official。
- 把 degraded route 写成 full success。
- 把 workbench / preview 写成平台内已执行。
- 把系统能力清单直接当经营结论。

## 排版要求

报告排版必须满足：
- 一屏内先看到结论，再看到问题，再看到动作。
- 标题层级不超过三级。
- 一条发现尽量 3 行内讲清。
- 一条行动尽量 6 行内讲清。
- 正文不使用大段原始表格或原始 JSON。
- 证据路径、脚本路径、路由路径只放在证据层或附录，不放在正文开头。

## 生成顺序要求

生成器必须按以下顺序工作：
1. 汇总 live route 数据和本地证据。
2. 识别 route 健康度。
3. 从稳定底层 route 提炼事实。
4. 从事实形成判断。
5. 从判断生成动作候选。
6. 按优先级排序动作。
7. 补齐盲区、降级说明和边界声明。
8. 最后填入模板并自评分。

禁止反过来先拿 route 输出直接拼正文。

## 交付门槛

一份正式可交付的 WIKA 运营报告，最低必须满足：
- 有执行摘要
- 有核心发现
- 有关键问题
- 有优先行动
- 有人工接手说明
- 有盲区说明
- 有路由健康与证据约束说明
- 有边界声明
- 没有把 derived 写成 official
- 没有把 degraded 写成 full success

达不到上述门槛的报告，只能算调试产物，不能发给业务方。
