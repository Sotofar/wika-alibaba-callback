# WIKA 真实数据源清单与优先级计划

## 当前已确认可用的数据源

### 1. 接入层状态接口

- `GET /health`
- `GET /integrations/alibaba/auth/debug`

用途：

- 只能确认服务和 token 状态
- 不能直接提供经营分析数据

## 当前尚未打通的业务数据源

### A. 官方接口

当前状态：

- 现有项目和已验证返回中，尚未发现店铺概览、产品表现、订单分析的业务读取实现
- 当前代码里真实调用的外部接口只有 token create / refresh

第一优先级建议：

- 继续以官方接口为优先
- 优先查证是否存在以下三类接口：
  - 店铺概览 / 经营概览
  - 产品列表 / 产品表现
  - 订单列表 / 订单分析

当前标记：

- `接口未找到（就当前项目与已验证结果而言）`

### B. 页面请求接口

当前状态：

- 后台页面理论上可见，但当前项目中未沉淀任何网络请求抓包结果
- 仓库中没有可靠的页面请求接口清单

第一优先级建议：

- 如果官方接口不能覆盖，优先抓后台“概览页”和“产品表现页”的页面请求
- 明确区分：
  - 页面接口是否只读
  - 数据字段是否稳定
  - 是否依赖会话 Cookie 而非 OAuth token

当前标记：

- `页面可见待接入`

### C. 后台导出文件

当前状态：

- 当前项目里没有导出文件读取逻辑

第一优先级建议：

- 如果页面支持导出“店铺概览 / 产品表现 / 订单数据”报表，可作为稳妥兜底
- 适合先做日报/周报

当前标记：

- `未接入`

### D. 浏览器自动化只读采集

当前状态：

- 当前项目里没有专门的数据采集脚本

第一优先级建议：

- 仅在前 3 条都不可行时使用
- 优先采集：
  - 概览卡片指标
  - 趋势图数字
  - 产品表现列表

当前标记：

- `兜底方案，未接入`

## 本轮优先打通顺序

1. 店铺整体概览
2. 产品表现
3. 订单分析
4. 广告与关键词

## 当前阶段结论

基于项目现状，当前最真实的判断是：

- 现有 token 能力已经具备继续向下扩展的前提
- 但真正的业务数据源仍需逐个识别与验证
- 第一优先级不是继续改授权，而是确定 `overview` 和 `products` 的可读业务源
